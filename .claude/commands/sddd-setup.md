対象プロジェクトに仕様書駆動開発（SdDD）の環境をセットアップします。

## 手順

### Step 1: 対象プロジェクトの確認

発注者に対象プロジェクトのパスを確認する。
カレントディレクトリが対象なら「このディレクトリにセットアップしますか？」と聞く。

### Step 2: 現状スキャン

対象プロジェクトの以下を調べて報告する:

- `CLAUDE.md` が既に存在するか
- `docs/` ディレクトリが存在するか
- `.claude/commands/` が存在するか
- `task.md` が存在するか
- プロジェクト内の既存ドキュメント（`.md` ファイル、README等）

報告例:
```
対象: /path/to/project
- CLAUDE.md: あり（既存の内容あり）
- docs/: なし
- .claude/commands/: なし
- 既存ドキュメント: README.md, docs/旧仕様.md（2件）
```

### Step 3: 発注者に個別に確認

以下を**個別に**確認する。まとめて聞かない:

1. **CLAUDE.md の導入方式**

   以下の2つの方式から選んでもらう:

   ```
   CLAUDE.md へのSdDDルール導入方式を選んでください。

   【A】参照方式（おすすめ）
     CLAUDE.md に「docs/sddd/rules.md を読め」の数行だけ追記。
     ✅ 既存のCLAUDE.mdをほぼ汚さない
     ✅ SdDDルールの更新がrules.md側だけで済む
     ⚠️ AIが毎セッション開始時にrules.mdを1回読みに行く手間が発生する

   【B】直接記載方式
     SdDDルール全体をCLAUDE.mdに追記（または新規作成）。
     ✅ CLAUDE.mdだけで完結。AIが即座にルールを把握できる
     ⚠️ CLAUDE.mdが長くなる（約200行追加）
     ⚠️ SdDDルールを更新したとき、CLAUDE.mdも手動で反映が必要
   ```

   - 既存CLAUDE.mdがない場合も同様に聞く（Aなら数行のCLAUDE.mdを新規作成、Bならフルのclaude.mdを新規作成）

2. **既存ドキュメントについて**（ある場合のみ）
   - 「既存のドキュメントが見つかりました。これを元に仕様書（docs/spec/）にまとめましょうか？新しいものほど重要として扱います」
   - Yes → Step 5 で取り込み処理を行う
   - No → 空のテンプレートを配置

3. **スラッシュコマンドについて**
   - 「SdDD用のスラッシュコマンド（/spec-sync, /debug, /kaigi 等）を配置しますか？」

### Step 4: ファイル配置

発注者の回答に応じてファイルを配置する。
テンプレートのソースは、このリポジトリの `templates/` ディレクトリから取得する。

#### 方式Aの場合（参照方式）

- `docs/sddd/rules.md` — `templates/CLAUDE.md` の内容をコピー（SdDDルール本体）
- `CLAUDE.md` — 以下の内容だけを追記:
  ```markdown
  ## 開発方式: 仕様書駆動開発（SdDD）

  このプロジェクトはSdDDで開発する。
  **セッション開始時に `docs/sddd/rules.md` を必ず読み、そのルールに従うこと。**
  ```

#### 方式Bの場合（直接記載方式）

- `CLAUDE.md` — `templates/CLAUDE.md` の内容を追記 or 新規作成

#### 共通

- `docs/SPEC.md` — 仕様書の目次
- `docs/requests.md` — 要望入力口
- `docs/request_log.md` — 要望対応履歴
- `docs/spec/` — 仕様書テンプレート群（01-06）
- `docs/handover/` — Agent引き継ぎ資料フォルダ
- `task.md` — タスク管理ファイル
- `.claude/commands/` — スラッシュコマンド群（Step 3で承認された場合）

### Step 5: 既存ドキュメントの取り込み（該当する場合のみ）

発注者が取り込みを希望した場合:

1. 既存ドキュメントをファイル更新日時の新しい順に読む
2. 内容を分類して対応する spec ファイルに反映:
   - プロジェクトの目的・概要 → `spec/01_概要.md`
   - 機能に関する記述 → `spec/02_機能仕様.md`
   - 画面・UIに関する記述 → `spec/03_画面設計.md`
   - DB・APIに関する記述 → `spec/04_データ設計.md`
   - 技術スタック・構成 → `spec/05_技術設計.md`
3. 取り込み元と取り込み内容を `spec/06_変更履歴.md` に記録
4. `SPEC.md` の目次を更新
5. 内容が矛盾する場合は新しいドキュメントを優先し、発注者に確認する

### Step 6: ダッシュボードの案内

```
進捗ダッシュボード（任意）を使うと、Agentの作業状況をリアルタイムで確認できます。

セットアップ:
  cd <spec-docs-driven-dev-templateのパス>/dashboard
  npm install && npm run build && npm link

使い方（別ターミナルで実行）:
  sdd-dashboard <プロジェクトルートの絶対パス>/task.md
```

### Step 7: 完了報告

セットアップ内容をまとめて報告し、次のステップを案内:

```
セットアップ完了しました。

配置したもの:
- CLAUDE.md（SdDDルール：参照方式 or 直接記載）
- docs/（仕様書テンプレート一式）
- .claude/commands/（スラッシュコマンド12個）
- task.md（タスク管理）

次のステップ:
- /sddd で開発を始められます
- docs/requests.md に要望を書くか、直接会話で伝えてください
```

## 注意

- 発注者の許可なくファイルを上書きしない
- 既存ファイルを削除しない
- セットアップ中に実装作業を開始しない
