# フォルダ構成

> プロジェクトに導入するフォルダ構成と、各ファイルの責任範囲を定義する。

## 構成

```
docs/
  SPEC.md              <- 仕様の目次・全体像（指揮AIが最初に読む）
  requests.md          <- ユーザーが要望を書く場所（未対応のみ）
  request_log.md       <- 全リクエスト履歴（指揮AIが管理）
  development_env.md   <- 開発環境設定（必要に応じて）
  spec/
    01_概要.md         <- 何を作るか・誰のためか・解決する課題
    02_機能仕様.md     <- 機能一覧と各機能の詳細
    03_画面設計.md     <- 画面構成・遷移
    04_データ設計.md   <- DB設計・データ構造
    05_技術設計.md     <- 技術スタック・アーキテクチャ
    06_変更履歴.md     <- 仕様変更の経緯と理由
  handover/            <- Agent引き継ぎ資料
```

## 各ファイルの役割

| ファイル | 書く人 | 読む人 | 更新タイミング |
|:--|:--|:--|:--|
| `requests.md` | ユーザー | 指揮AI | ユーザーが要望を思いついたとき |
| `request_log.md` | 指揮AI | 指揮AI | 要望を受け付けたとき / 対応完了時 |
| `SPEC.md` | 指揮AI | 指揮AI / Agent | spec を更新したとき |
| `spec/01_概要.md` | 指揮AI | 指揮AI / Agent | プロジェクト立ち上げ時 / 方向性が変わったとき |
| `spec/02_機能仕様.md` | 指揮AI | Agent | 機能追加・変更のたび |
| `spec/03_画面設計.md` | 指揮AI | Agent | 画面追加・変更のたび |
| `spec/04_データ設計.md` | 指揮AI | Agent | テーブル追加・変更のたび |
| `spec/05_技術設計.md` | 指揮AI | Agent | 技術スタック決定時 / アーキテクチャ変更時 |
| `spec/06_変更履歴.md` | 指揮AI | 指揮AI | 既存仕様を変更したとき |
| `development_env.md` | 指揮AI | Agent | 開発環境の設定が変わったとき |
| `handover/*.md` | 実装Agent | 新Agent | Agent引き継ぎ時 |

## ポイント

### requests.md はユーザー専用の入力口

未対応の要望だけを書く場所。指揮AIが仕様書に反映した要望は、指揮AIが削除する。ユーザーは「書いて伝える」だけでよい。

### SPEC.md は目次

全ての spec ファイルの概要と最終更新日を記載する。指揮AIはまず SPEC.md を読み、今回の作業に関係する spec だけを選択的に開く。これによりトークンを節約する。

### handover/ は必要になってから作る

最初から用意する必要はない。Agentの引き継ぎが発生した時点で作成する。

### request_log.md は全履歴

requests.md が「今」の要望だけを持つのに対し、request_log.md は全リクエストの履歴を保持する。日付・内容・対応状況・反映先の spec を記録する。
