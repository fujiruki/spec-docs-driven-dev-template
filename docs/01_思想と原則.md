# 思想と原則

> 仕様書駆動開発（Spec-Driven Orchestrated Development）の基本思想を解説する。

## 前提：Claude Code の Agent 機能

本ワークフローは Claude Code の Agent 機能（バックグラウンドでコード生成・編集を行う独立エージェント）を前提としている。Agent を「実装担当」として使い、メインの Claude を「指揮AI」として使う。

## AI開発で起きる3つの問題

### 1. 文脈が消える

AIとのセッションが切れると、「何を作ろうとしていたか」が消える。次のセッションでは一から説明し直しになる。

### 2. 会話が止まる

AIがコードを書いている間、ターミナルが占有されて人間は何もできない。「ここも直してほしい」と思いついても、作業完了まで待つしかない。

### 3. 並列に進められない

複数の機能を頼みたくても、1つずつ順番に作らせるしかない。

## 解決策：仕様書 + 指揮AI + 実装Agent

これらの問題を一挙に解決するのが、仕様書駆動開発の構造である。

### 基本思想

1. **人間が伝えた要望を指揮AIが仕様書にまとめ、実装Agentに渡して開発する**
   - 人間は「こうしたい」と言うだけ。仕様の構造化はAIの仕事
2. **指揮AIは仕様整理と指揮に専念し、コードには一切触らない**
   - 指揮AIがコードを書かないから、会話が止まらない。思いついた瞬間に伝えられる
3. **セッションが切れても、仕様書と引き継ぎ資料を読めば文脈を復元できる**
   - 仕様書が「何を作ろうとしているか」の永続ストレージになる
4. **仕様書・引き継ぎ資料の管理はAI側の責任。人間の手間を最小限にする**
   - 人間がやるのは「要望を伝える」「確認に答える」「承認する」の3つだけ

### 仕様書が全ての土台

仕様書がしっかりしていれば、Agentは的確に動く。仕様書がなければ、何体Agentを立てても迷子になる。

仕様書は「Agentへの指示書」であり、「セッションの復元データ」であり、「Agentの引き継ぎ資料の基盤」でもある。

```
仕様書駆動開発 ── 何を作るかの管理
指揮AI + Agent体制 ── どう作るかの実行
```

この2つが組み合わさって初めて、「要望を伝えたら勝手に作られる開発」が成立する。

## 必要なもの

- Claude Code（Agent機能が使える環境）
- 仕様書駆動開発のフォルダ構成（`docs/spec/`）
- それだけ。特別なツールもサーバーも不要
