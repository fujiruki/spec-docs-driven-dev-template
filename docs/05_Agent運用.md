# Agent運用

> 実装Agentの分割・管理・引き継ぎに関するルールを定義する。

## 機能スコープ分割の原則

**レイヤー（フロント/バック）ではなく、機能で切る。**

1つの機能に関わるフロント・バック・API・DBは同じAgentが担当する。

```
NG - レイヤー分割:
  Agent-フロント: ログイン画面 + ダッシュボード画面
  Agent-バック:   認証API + 集計API

OK - 機能スコープ分割:
  Agent-認証:         ログイン画面 + 認証API + セッション管理
  Agent-ダッシュボード: ダッシュボード画面 + 集計API + グラフ描画
```

レイヤーで分けると、フロントとバックの仕様を知らないAI同士が別々に作ることになり、整合性が崩れる。機能で切れば、1つのAgentがフロントからバックまで一貫して持つので整合性が保たれる。

## 同時稼働数

PCスペックに応じて調整する。メモリやCPUの余裕を見ながら、指揮AIが稼働数を判断する。

## 生存方針：解放しない

Agentは機能完成後も解放しない。コンテキストを保持し続ける。

```
Agent-認証：ログイン機能を実装
  |
ユーザー：「ログインでエラー出る」
  |
指揮AI -> 同じAgent-認証に「このバグ直して」
  |
Agent-認証：自分が書いたコード全部覚えてる -> 即デバッグ
```

解放してしまうと、バグが見つかったときに一から文脈を説明し直すことになる。作った本人に直させるのが最も効率的。

## 劣化監視

Agentは活動量が増えると性能が劣化する。指揮AIが以下の兆候を監視する。

| 兆候 |
|------|
| 同じミスを2回以上繰り返す |
| 指示した内容を漏らす・忘れる |
| 修正が場当たり的になる（根本対処しなくなる） |
| 関係ないファイルを触り始める |
| 応答が冗長になる・同じ説明を繰り返す |

**2つ以上該当したら引き継ぎを提案する。** 最終判断はユーザー。

## 引き継ぎ手順

1. 指揮AIが「そろそろ引き継ぎしましょうか」と提案
2. ユーザーが承認
3. 劣化Agentが引き継ぎ資料を `docs/handover/` に作成
   - 何を作ったか
   - 未完了の作業
   - 既知の問題
   - 設計判断の経緯（specに書いてないもの）
4. Agent解放
5. 新Agent起動（引き継ぎ資料 + 該当specを読ませる）

仕様書駆動開発の思想がそのまま活きる。「書いてあるから復元できる」。セッションの復元だけでなく、Agentの復元にも使える。

## 実行方式

| 種別 | 方式 | 理由 |
|------|------|------|
| 実装・修正・デバッグ | バックグラウンド | 会話を止めない |
| 調査・質問への回答 | フォアグラウンド | 結果を待つ必要がある |

バックグラウンド実行が基本。指揮AIは常に会話可能な状態を保ち、ユーザーが思いついた要望をいつでも受け付けられるようにする。

会話をブロックする長時間のフォアグラウンド実行は禁止。
