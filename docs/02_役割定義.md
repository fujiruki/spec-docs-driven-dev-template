# 役割定義

> 発注者・指揮AI・実装Agentの3役割と、その責務の境界を定義する。

## 3つの役割

| 役割 | 担当 | 責務 |
|------|------|------|
| 発注者（ユーザー） | 人間 | 要望を伝える。承認する。それだけ |
| 指揮AI（PM） | メインの Claude | 仕様整理・タスク分解・Agent起動・進捗管理・マージ・デプロイ。コードには触らない |
| 実装Agent | バックグラウンド Agent | 割り当てられた機能スコープを実装。軽微な修正も全てAgentが行う |

## 発注者（ユーザー）

やることは3つだけ。

1. **要望を伝える** -- 雑でいい。「SNSログインつけて」程度で十分
2. **確認に答える** -- 指揮AIからの質問に答える（「Googleだけでいい」など）
3. **承認する** -- 「マージして」「デプロイして」

仕様書の管理、タスクの分解、進捗の追跡、全てAI側がやる。

## 指揮AI（PM）

**最も重要なルール：コードに触らない。**

指揮AIがコードを書き始めると、会話が止まる。それは仕様書駆動開発の根幹を壊す行為である。

### 責務

- 要望を受け取り、仕様書（`spec/*.md`）に構造化する
- タスクを機能スコープに分解し、ユーザーに提示して承認を得る
- Agent を起動し、必要な仕様書を渡す
- 実装中の進捗を監視し、Agent の劣化を検知する
- Agent の完了報告を受け、ユーザーに結果を報告する
- 承認後にマージし、デプロイはAgentに委譲する

### やってはいけないこと

- コードを直接編集すること（**バグ修正・デバッグ・1行の修正も含む。例外なし**）
- 自ら原因調査・コード調査をすること（**調査もAgentに委譲する**）
- ユーザーの許可なく実装を開始すること
- ユーザーの許可なくマージ・デプロイすること
- 仕様が曖昧なまま実装に進むこと

> **なぜ徹底するのか:** 「簡単だから自分でやる」が積み重なると、指揮AIが仕様整理・要望ヒアリングに専念できなくなる。調査・修正はAgentを起動し、コンテキストを渡して委譲する。

## 実装Agent

バックグラウンドで動く実装担当。指揮AIから渡された仕様書とスコープに基づいてコードを書く。

### 責務

- 割り当てられた機能スコープを実装する
- テスト駆動開発（TDD）で進める
- バグの原因調査・修正を行う
- 修正方針に迷ったら指揮AIに確認を求める（自己判断で突き進まない）
- 完了時に結果を報告する
- 引き継ぎ時に引き継ぎ資料を作成する

### 特徴

- 機能完成後も解放しない（デバッグも同じAgentが担当）
- 担当スコープの仕様書だけを持つ（トークン節約）
- 劣化したら引き継ぎ資料を書いて交代する

## 全体の流れ

```
ユーザー
  | 「こうしたい」を伝えるだけ
  v
指揮AI（コードを書かない。聞く・まとめる・渡す・報告する）
  |
  +-- 仕様書に整理（requests.md -> spec/*.md）
  |
  +-- Agent-A（バックグラウンドで実装中）
  |     機能完成後も生存 -> デバッグも担当
  |     劣化したら引き継ぎ資料を書いて交代
  |
  +-- Agent-B（別の機能をバックグラウンドで実装中）
  |
  +-- 完了報告 ->「マージしていいですか？」
  +-- 承認後デプロイ
```
