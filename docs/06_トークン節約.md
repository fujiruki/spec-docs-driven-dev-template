# トークン節約

> 仕様書の読み込みを最小限にし、トークン消費を抑える仕組みを解説する。

## 問題

プロジェクトが大きくなると、仕様書のファイル数・文量も増える。毎回全ファイルを読み込むとトークンを大量に消費し、コンテキストウィンドウを圧迫する。

## 解決策：SPEC.md を目次として機能させる

### SPEC.md の書き方

```markdown
# SPEC.md

## 概要
タスク管理Webアプリケーション。チームでタスクを共有・管理する。

## 仕様書一覧

| ファイル | 最終更新 | 概要 |
|:--|:--|:--|
| spec/01_概要.md | 2025-01-15 | プロジェクトの目的・対象ユーザー・解決する課題 |
| spec/02_機能仕様.md | 2025-02-20 | タスクCRUD、カテゴリ、検索、通知機能 |
| spec/03_画面設計.md | 2025-02-18 | 5画面構成。ダッシュボード・一覧・詳細・設定・ログイン |
| spec/04_データ設計.md | 2025-02-20 | tasks, users, categories, notifications テーブル |
| spec/05_技術設計.md | 2025-01-15 | React + Vite / PHP API / MySQL |
| spec/06_変更履歴.md | 2025-02-20 | 通知機能追加に伴うDB変更 |
```

### 指揮AIの読み込みルール

1. セッション開始時に `SPEC.md` だけを読む
2. 目次の概要から、今回の作業に関係する spec を判断する
3. **関係する spec だけを開く**（全部は読まない）

例：「通知機能を修正したい」という要望なら、`02_機能仕様.md` と `04_データ設計.md` だけを開けばよい。`03_画面設計.md` や `05_技術設計.md` は読まなくてよい。

### Agentへの仕様書の渡し方

Agentには担当スコープに関係する spec だけを渡す。

```
Agent-通知機能:
  読む: spec/02_機能仕様.md, spec/04_データ設計.md
  読まない: spec/01_概要.md, spec/03_画面設計.md, spec/05_技術設計.md
```

全 spec を渡すと、関係のない情報でトークンを浪費し、Agentの判断精度も下がる。

## まとめ

- SPEC.md に各 spec ファイルの最終更新日と概要を記載する
- 指揮AIは今回の作業に関係する spec だけを開く
- Agentには担当スコープの spec だけを渡す
- この3つを守るだけでトークン消費を大幅に削減できる
