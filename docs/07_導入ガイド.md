# 導入ガイド

> 既存プロジェクトへの段階的導入と、ゼロから始める場合の手順を解説する。

## 前提条件

- Claude Code がインストールされていること
- Agent機能（バックグラウンドAgent）が使えること

## A. ゼロから始める場合

### 1. フォルダ構成を作る

```bash
mkdir -p docs/spec docs/handover
```

### 2. テンプレートを配置する

`templates/` ディレクトリにテンプレートファイルが用意されている場合は、コピーして使う。

```bash
# 仕様書テンプレート
cp templates/docs/SPEC.md docs/SPEC.md
cp templates/docs/requests.md docs/requests.md
cp templates/docs/request_log.md docs/request_log.md
cp -r templates/docs/spec/ docs/spec/

# CLAUDE.md（指揮AIのルール）
cp templates/CLAUDE.md CLAUDE.md

# タスク管理ファイル（ダッシュボード連携）
cp templates/task.md task.md

# スラッシュコマンド（Claude Code用）
cp -r templates/.claude .claude
```

テンプレートがない場合は、以下の最小構成から始める。

**docs/SPEC.md:**
```markdown
# SPEC.md

## 概要
（プロジェクトの1行説明）

## 仕様書一覧

| ファイル | 最終更新 | 概要 |
|:--|:--|:--|
| spec/01_概要.md | （日付） | （概要） |
```

**docs/requests.md:**
```markdown
# Requests

<!-- 要望をここに書く。反映済みのものは指揮AIが削除する -->
```

**docs/request_log.md:**
```markdown
# Request Log

| 日付 | 内容 | 状況 | 反映先 |
|:--|:--|:--|:--|
```

### 3. CLAUDE.md を設定する

プロジェクトルートの `CLAUDE.md` に、指揮AIの行動ルールを記載する。

```markdown
# CLAUDE.md

## 開発ワークフロー

このプロジェクトは仕様書駆動開発で進める。

### セッション開始時
1. `docs/SPEC.md` を読む（目次・概要のみ）
2. `docs/requests.md` を読む（未対応の要望があるか確認）
3. `docs/handover/` を確認（引き継ぎがあるか）
4. 今回の作業に関係する spec だけを選択的に読む

### ルール
- 指揮AIはコードに触らない。実装は全てAgentに任せる
- ユーザーの許可なく実装・マージ・デプロイを開始しない
- 仕様が曖昧なら実装に進まず確認を返す
- 実装はバックグラウンドで実行し、会話を止めない
```

### 4. ダッシュボードCLIをインストールする（任意）

進捗ダッシュボードを使うと、Agentの作業状況をリアルタイムで確認できます。

```bash
# spec-docs-driven-dev-template リポジトリの dashboard ディレクトリで実行
cd <spec-docs-driven-dev-templateのパス>/dashboard
npm install && npm run build && npm link
```

インストール後、プロジェクトのディレクトリで別のターミナルから起動します：

```bash
sdd-dashboard task.md
```

### 5. 開発を開始する

Claude Code で `/sddd` を実行すると、SdDDの開始ガイドが表示されます。
ガイドに従って要望を伝えれば、指揮AIが仕様整理からタスク分解まで進めてくれます。

### 6. 最初の要望を伝えてみる

```
ユーザー：「タスク管理アプリを作りたい。タスクの追加・完了・一覧表示ができればいい」
```

指揮AIが以下を実行する：
1. `spec/01_概要.md` に目的を整理
2. `spec/02_機能仕様.md` に機能を整理
3. `SPEC.md` の目次を更新
4. `request_log.md` に記録
5. タスク分解を提示し、承認を求める

承認後、Agentがバックグラウンドで実装を開始する。

## B. 既存プロジェクトへの段階的導入

一気に移行しない。段階的に進める。

### Step 1: 最小構成を追加

```bash
mkdir -p docs/spec docs/handover
touch docs/SPEC.md docs/requests.md docs/request_log.md
```

既存の docs がある場合はそのまま残す。

### Step 2: SPEC.md を目次として作成

既存ドキュメントへのリンクを含めてよい。

```markdown
# SPEC.md

## 概要
（既存プロジェクトの説明）

## 既存ドキュメント
- docs/README.md -- 元々あったドキュメント
- docs/api.md -- API仕様

## 仕様書一覧

| ファイル | 最終更新 | 概要 |
|:--|:--|:--|
```

### Step 3: 新しい要望から spec に整理していく

既存の全ドキュメントを spec/ に移行する必要はない。新しい要望が来たときに、対応する spec を作っていけばよい。

### Step 4: CLAUDE.md にルールを追記

既存の CLAUDE.md に仕様書駆動開発のルールを追記する。

### Step 5: handover/ は必要になったら作る

Agentの引き継ぎが発生するまで、空のディレクトリのまま置いておけばよい。

### Step 6: ダッシュボードとスラッシュコマンドを導入

```bash
# タスク管理ファイルとスラッシュコマンドをコピー
cp templates/task.md task.md
cp -r templates/.claude .claude

# ダッシュボードCLIをインストール（任意）
cd <spec-docs-driven-dev-templateのパス>/dashboard
npm install && npm run build && npm link
```

導入後、Claude Code で `/sddd` を実行して開発を開始できます。

## よくある疑問

### Q: spec ファイルは6つ全部必要？

不要。プロジェクトの規模に合わせて必要なものだけ作ればよい。小さなプロジェクトなら `01_概要.md` と `02_機能仕様.md` だけで十分。

### Q: requests.md は必須？

プロンプトで直接要望を伝えるなら不要。ただし、思いついたときにメモしておきたい場合に便利。

### Q: 1人開発でもAgent体制は必要？

仕様書管理だけでも十分に価値がある。Agent体制は「会話のテンポを止めたくない」「複数機能を並列に進めたい」場合に導入すればよい。
