# ワークフロー

> 要望の受付からデプロイまでの6フェーズを定義する。

## 要望の入力方法

### A. ファイル経由

`docs/requests.md` に要望を書き、指揮AIに反映を依頼する。

```markdown
<!-- requests.md の例 -->
- ログイン機能にGoogleアカウント対応を追加してほしい
- ダッシュボードに月次売上グラフを表示したい
```

### B. プロンプト直接

指揮AIとの会話で「こういう機能ほしい」と直接伝える。

どちらの場合も指揮AIの処理は同じ。

## 6つのフェーズ

### Phase 1: 受付

- 要望を受け取る（会話 or `requests.md`）
- `request_log.md` に記録する

```
ユーザー：「SNSログインつけて。あとダッシュボードにグラフ出したい」
指揮AI ：request_log.md に記録 -> Phase 2 へ
```

### Phase 2: 仕様整理

- 該当する `spec/*.md` を更新（なければ新規作成）
- `SPEC.md` の目次を更新
- **仕様が曖昧なら実装に進まず、確認を返す**
- 既存仕様の変更の場合は `06_変更履歴.md` に記録

```
指揮AI：仕様確認します。
  - SNSログイン：Googleだけ？LINEも？
  - グラフ：どのデータから月次集計？

ユーザー：「Googleだけ。売上データから」

指揮AI：了解。spec/02_機能仕様.md を更新します。
```

**仕様未整理のまま実装に突入してはいけない。** 曖昧さが残っている場合は、必ずユーザーに確認する。

### Phase 3: タスク分解・承認

指揮AIが以下を判断し、提示する。

- いくつの機能スコープに分かれるか
- 各スコープの実装内容
- Agent構成（新規起動 or 既存Agentへの追加指示）

```
指揮AI：2つのAgentで進めます。
  - Agent-認証：Googleログイン追加（画面 + API + セッション管理）
  - Agent-ダッシュボード：売上グラフ表示（画面 + 集計API + グラフ描画）
  よろしいですか？

ユーザー：「おっけ」
```

**ユーザーの承認を得てから Phase 4 へ進む。許可なく実装を開始しない。**

### Phase 4: 実装（バックグラウンド）

- Agent はバックグラウンドで実行（指揮AIは会話可能な状態を維持）
- ユーザーは次の要望を伝えてよい（Phase 1 に戻る）
- 実装はTDDで進める

```
ユーザー：「あ、あとヘッダーの色もうちょい暗くして」
指揮AI ：了解、Agentに回します。他にありますか？
```

会話のテンポが止まらないことが、この体制の最大の利点。

### Phase 5: 完了・レビュー・マージ

- Agent完了時、指揮AIが結果を報告
- ユーザーの承認後、メインブランチにマージ
- `request_log.md` の対応状況を「実装完了」に更新

### Phase 6: デプロイ

- ユーザーの承認後、指揮AIがデプロイを実行

## セッション中の仕様変更

対話中に「もっとこうして」と言われたら：

1. 指揮AIが該当する `spec/*.md` を更新
2. `06_変更履歴.md` に追記
3. `request_log.md` に記録
4. 実装中のAgentがあれば、追加指示を送る

仕様変更は日常的に起こる。フローの中に組み込まれているので、特別な手続きは不要。

## 指揮AIのセッション開始時の行動

1. `docs/SPEC.md` を読む（目次・概要のみ）
2. `docs/requests.md` を読む（未対応の要望があるか確認）
3. `docs/handover/` を確認（前セッションからの引き継ぎがあるか）
4. 今回の作業に関係する spec だけを選択的に読む
