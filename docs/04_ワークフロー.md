# ワークフロー

> 要望の受付からデプロイまでの6フェーズを定義する。

## 要望の入力方法

### A. ファイル経由

`docs/requests.md` に要望を書き、指揮AIに反映を依頼する。

```markdown
<!-- requests.md の例 -->
- ログイン機能にGoogleアカウント対応を追加してほしい
- ダッシュボードに月次売上グラフを表示したい
```

### B. プロンプト直接

指揮AIとの会話で「こういう機能ほしい」と直接伝える。

どちらの場合も指揮AIの処理は同じ。

## 6つのフェーズ

### Phase 1: 受付

- 要望を受け取る（会話 or `requests.md`。どちらも同じ扱い）

```
ユーザー：「SNSログインつけて。あとダッシュボードにグラフ出したい」
指揮AI ：了解。仕様を詰めましょう -> Phase 2 へ
```

### Phase 2: 仕様詰め

- 要望を仕様として対話で詰める
- **仕様が曖昧なら実装に進まず、確認を返す**

```
指揮AI：仕様確認します。
  - SNSログイン：Googleだけ？LINEも？
  - グラフ：どのデータから月次集計？

ユーザー：「Googleだけ。売上データから」

指揮AI：了解、仕様が固まりました。
```

**仕様未整理のまま実装に突入してはいけない。** 曖昧さが残っている場合は、必ずユーザーに確認する。

### Phase 3: 仕様確定・記録

仕様が固まったら、以下を実行する:

1. `request_log.md` に追記（日付・内容・反映先spec）
2. 該当する `spec/*.md` を更新（なければ新規作成）
3. `SPEC.md` の目次を更新
4. 既存仕様の変更の場合は `06_変更履歴.md` に記録
5. `requests.md` から読み込み済みの内容をクリア

```
指揮AI：仕様を確定しました。
  - request_log.md に記録しました
  - spec/02_機能仕様.md を更新しました
  - requests.md の該当内容をクリアしました
```

### Phase 4: 実装（バックグラウンド）

仕様が確定したら、タスク分解とAgent割り振りはAIが自律的に行う。発注者の承認は不要。

#### 4-1. task.md 作成とダッシュボード案内

**Agent起動前に必ず実行する。**

1. **複数の要望がある場合は、すべての仕様確定を終えてから**タスクに分解する（1つ目の要望だけでAgent起動しない）
2. 仕様をタスクに分解し、プロジェクトルートに `task.md` を作成（チェックボックス付き）
3. ユーザーに「別のターミナルで `sdd-dashboard task.md` を開くと進捗を確認できます」と案内
4. **案内後、ユーザーの確認を待たずにAgentを起動する**（ダッシュボードを開くかどうかはユーザーの任意）

```markdown
<!-- task.md の例 -->
# 写真ドラッグ&ドロップ機能

## Agent-ViewModel（TDD）
- [ ] テスト作成: movePhotoToStep
- [ ] テスト作成: movePhotoToUnassigned
- [ ] テスト作成: reorderPhotosInStep
- [ ] テスト実行→失敗確認
- [ ] 3メソッド実装
- [ ] テスト実行→全パス確認

## Agent-UI
- [ ] DndContext再設計
- [ ] 未割り当てプール常時表示
- [ ] StepCard写真ドラッグ対応
- [ ] DragOverlay実装
- [ ] TouchSensor追加
```

#### 4-2. Agent起動と実行

- Agent はバックグラウンドで実行（指揮AIは会話可能な状態を維持）
- **Agentはサブタスク完了ごとに `task.md` のチェックボックスを更新する**
- ユーザーは次の要望を伝えてよい（Phase 1 に戻る）
- 実装はTDDで進める

```
指揮AI ：task.md を作成しました。Agentを起動します。
         💡 別のターミナルで `sdd-dashboard task.md` を開くと進捗を確認できます。
ユーザー：「あ、あとヘッダーの色もうちょい暗くして」
指揮AI ：了解、Agentに回します。他にありますか？
```

会話のテンポが止まらないことが、この体制の最大の利点。

### Phase 5: 完了・レビュー・マージ

- Agent完了時、指揮AIが結果を報告
- ユーザーの承認後、**Agentがコミット・マージを実行**（指揮AIは実行しない）
- `request_log.md` の対応状況を「実装完了」に更新

### Phase 6: デプロイ

- ユーザーの承認後、**指揮AIはデプロイAgentを起動**する（指揮AI自身はデプロイを実行しない）
- デプロイAgentがデプロイを実行し、成否を確認し、`task.md` を更新する
- 指揮AIはAgentからの完了報告をユーザーに伝える

## セッション中の仕様変更

対話中に「もっとこうして」と言われたら：

1. 指揮AIが該当する `spec/*.md` を更新
2. `06_変更履歴.md` に追記
3. `request_log.md` に記録
4. 実装中のAgentがあれば、追加指示を送る

仕様変更は日常的に起こる。フローの中に組み込まれているので、特別な手続きは不要。

## 指揮AIのセッション開始時の行動

1. `docs/SPEC.md` を読む（目次・概要のみ）
2. `docs/requests.md` を読む（未対応の要望があるか確認）
3. `docs/handover/` を確認（前セッションからの引き継ぎがあるか）
4. 今回の作業に関係する spec だけを選択的に読む
