仕様書駆動開発（SdDD）を開始します。

## Step 0: プロジェクトのSdDD環境チェック

まず現在のプロジェクトにSdDD環境があるか確認する:

- `docs/SPEC.md` が存在するか
- `docs/spec/` が存在するか
- `docs/requests.md` が存在するか
- `task.md` が存在するか

**すべて存在する場合** → Step 0 をスキップし「セッション開始時の行動」へ進む。
**不足がある場合** → 以下のプロジェクトセットアップを実行する。

### プロジェクトセットアップ（初回のみ）

SdDDテンプレートリポジトリ（グローバルCLAUDE.mdに記載されたパス）の `templates/` からファイルを配置する。

1. **CLAUDE.md の導入方式を確認**

   ```
   このプロジェクトのCLAUDE.md にSdDDルールを導入します。方式を選んでください。

   【A】参照方式（おすすめ）
     CLAUDE.md に「docs/sddd/rules.md を読め」の数行だけ追記。
     ✅ 既存のCLAUDE.mdをほぼ汚さない
     ✅ SdDDルールの更新がrules.md側だけで済む
     ⚠️ AIが毎セッション開始時にrules.mdを1回読みに行く手間が発生する

   【B】直接記載方式
     SdDDルール全体をCLAUDE.mdに追記（または新規作成）。
     ✅ CLAUDE.mdだけで完結。AIが即座にルールを把握できる
     ⚠️ CLAUDE.mdが長くなる（約200行追加）
     ⚠️ SdDDルールを更新したとき、CLAUDE.mdも手動で反映が必要
   ```

2. **方式Aの場合**: `docs/sddd/rules.md` に `templates/CLAUDE.md` の内容をコピー。プロジェクトの CLAUDE.md には参照の数行だけ追記
3. **方式Bの場合**: プロジェクトの CLAUDE.md に `templates/CLAUDE.md` の内容を追記

4. **共通ファイルを配置**:
   - `docs/SPEC.md` — 仕様書の目次
   - `docs/requests.md` — 要望入力口
   - `docs/request_log.md` — 要望対応履歴
   - `docs/spec/` — 仕様書テンプレート群（01-06）
   - `docs/handover/` — Agent引き継ぎ資料フォルダ
   - `task.md` — タスク管理ファイル

5. **既存ドキュメントの取り込み確認**（既存の `.md` ファイルがある場合のみ）:
   - 「既存のドキュメントが見つかりました。仕様書に取り込みましょうか？」
   - Yes → 内容を分類して `spec/*.md` に反映
   - No → 空のテンプレートを配置

6. セットアップ完了を報告し、そのまま「セッション開始時の行動」へ進む

---

## セッション開始時の行動

以下を順に確認し、状況に応じた声かけを行う。**すべて確認し終わってから発注者に報告する。**

1. `docs/SPEC.md` を読む（目次・概要の把握）
2. `docs/requests.md` を読む（未対応の要望があるか確認）
3. `docs/handover/` を確認（前セッションからの引き継ぎがあるか）
4. `task.md` を確認（前回の作業途中のタスクがあるか）
5. 今回の作業に関係する `docs/spec/` 内のファイルだけを選択的に読む

### 確認結果に応じた声かけ

上記の確認結果を**まとめて1回で報告**する。以下の優先順位で判断する:

**引き継ぎ資料がある場合:**
```
前回の引き継ぎ資料があります。読み込みました。
[引き継ぎ内容の要約]
続きから進めますか？
```

**task.md に未完了タスクがある場合:**
```
前回のタスクが残っています。
[未完了タスクの要約: X件中Y件完了]
続きから進めますか？それとも別の作業をしますか？
```

**requests.md に未対応の要望がある場合:**
```
未対応の要望がX件あります。
[要望の概要]
仕様に反映して進めましょうか？
```

**すべてクリーンな場合（初回セットアップ直後 or 前回すべて完了済み）:**
```
プロジェクトの現状を把握しました。
[SPEC.mdの概要]
何を作りますか？または改善したいことはありますか？
```

## ダッシュボード（任意）

実装が始まったら、**別のターミナル**で進捗ダッシュボードを表示できます：

```bash
sdd-dashboard <プロジェクトルートの絶対パス>/task.md
```

ダッシュボードを開くかどうかはユーザーの任意です。指揮AIは案内だけ行い、確認を待たずにAgentを起動します。

## 主要コマンド

| コマンド | 用途 |
|---------|------|
| `/spec-sync` | `requests.md` の要望を仕様書に反映 |
| `/kakunin` | 実装前に止まって確認 |
| `/soudan` | 相談（コードに触らない） |
| `/debug` | エラー修正（根本原因追求） |
| `/kaigi` | AI専門家会議（多角的な議論） |

## ワークフロー

```
Phase 1: 受付 → Phase 2: 仕様詰め → Phase 3: 仕様確定・記録
→ Phase 4: task.md作成 → ダッシュボード案内 → Agent実装
→ Phase 5: レビュー → Phase 6: デプロイ（Agentが実行・成否確認）
```

### Phase 4 チェックリスト（指揮AI必須）

1. **複数の要望がある場合は、すべての仕様確定を終えてから**タスク分解する
2. `task.md` をプロジェクトルートに作成（全要望分のタスクを記載）
3. 「別ターミナルで `sdd-dashboard <絶対パス>/task.md` で進捗確認できます」と案内
4. **案内後、確認を待たずにAgentを起動**（ダッシュボードはユーザーの任意）
5. Agentにはサブタスク完了ごとに `task.md` を更新する指示を含める
